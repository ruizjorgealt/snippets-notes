Types Of Automated Testing

Unit Testing

Testing at the unit level; specific class.
Unit testing tests the smallest unit of code in isolation.

Integration

Instead of testing individual units, integration tests how these units interact with other. This is still at the code level.
Integration testing tests units together, but still focuses on the code. Not working from the perspective of a user.

Blackbox Automated Testing (BAT)

BATs test the system from the perspective of the user. This will become automated. Automation becomes hard, thus the framework helps a lot.

Why Do It?

You are basically adding stuff every week, this makes sure that we don’t break things. Things may work without us knowing, the user will notice.

Absolute Requirements

BAT can serve as an abosulte requirement.

"This test explains exactly what I want and what you agreed to"

Leverage

BATs test much more production code with fewer lines of BAT code.

Common Failure Points

  • Recorded brittle tests
    • People end up using a recording tool such as Selenium IDE
    • Time consuming and repetitive
  • Not building a framework
  • Writing tests like code
    • Tests need to be simplified even if using programming language
    • Complexity make them hard to maintain and brittle as well

Our Approach

  • Seperate automation framework
    • Tests don’t directly work against the web app
    • TESTS → FRAMEWORK → WEB APP
  • Simple tests
    • The tests aren’t going to be doing a lot
    • Tests should be in very simple terms and language
    • Easy to modify
  • Tests drive creation of framework
    • The test drives the creation of the framework
